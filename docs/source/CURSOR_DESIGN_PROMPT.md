# DESIGN COMPLETION PROMPT — hook.press

К этому промпту приложен **Master Prompt hook.press** — используй его как источник истины по составу разделов, вкладок, кнопок и экранов (разделы 7–22: домены backend и требования к Web/Flutter). Также используй текущее состояние репозитория `apps/web` как есть.

## Проблема

Текущий UI пустой, не заполнен, выглядит как заготовка, а не рабочий продукт: не хватает вкладок, кнопок, состояний, визуальной идентичности. Нужно довести дизайн до уровня полноценной музыкальной платформы (уровня продукта, которым реально пользуются: сравнимо по плотности и продуманности с Spotify for Artists, SoundCloud, Bandcamp, DistroKid, Splice) — не макет, а рабочий интерфейс, подключённый к реальному backend API.

## Не делать

- Не создавай декоративные страницы без данных из API — если экран показывает список/карточки, он должен реально дёргать backend, а если данных ещё нет — показывать честный empty state, а не заглушку "Coming soon".
- Не копируй один в один визуальный стиль Spotify/SoundCloud (цвета, логотип, шрифты) — ориентируйся на них только по уровню плотности функциональности и качеству UX-паттернов. Дизайн hook.press должен быть самостоятельным и узнаваемым.
- Не используй дефолтные AI-паттерны: тёплый кремовый фон с терракотовым акцентом, чёрный фон с одним кислотным акцентом, газетный layout с hairline-линиями — если это не осознанный выбор под бренд hook.press.
- Не ломай уже работающую backend-интеграцию существующих экранов.

## Дизайн-направление

hook.press — платформа, где артист проходит полный путь от написания текста и генерации демо до релиза, продвижения и заработка на маркете услуг. Это рабочий инструмент музыканта, а не развлекательный стриминг-сервис. Ключевая эмоция: студия/пульт/рабочее место продюсера — сфокусированность, контроль, ощущение прогресса.

Перед началом кодирования выполни этап планирования дизайна (обязательно зафиксировать в `docs/DESIGN_SYSTEM.md`):

1. **Палитра** — 4–6 именованных hex-значений, обоснованных смыслом продукта (не дефолтный кремовый/терракотовый и не голый чёрный с одним акцентом без обоснования).
2. **Типографика** — display-шрифт с характером (для заголовков, названий треков, чисел прогресса) + body-шрифт для текста + моноширинный/utility-шрифт для технических данных (BPM, длительность, статусы, ID). Обоснуй пару шрифтов исходя из бренда.
3. **Layout-концепция** — как организованы основные экраны: сайдбар/топбар, плотность карточек, соотношение списков и деталей. ASCII-wireframes для ключевых экранов (дашборд, студия, офис/релиз, маркет, лента).
4. **Сигнатурный элемент** — один узнаваемый визуальный приём, который будет присутствовать сквозь весь продукт (например: визуализация waveform как элемент навигации, индикатор прогресса как физический "фейдер", карта рифм как визуальный граф и т.п.) — выбери то, что реально отражает суть продукта.

Проведи самокритику плана: если что-то похоже на типовой шаблон — перепиши и обоснуй, что изменено и почему. Только после этого приступай к реализации.

## Обязательный набор экранов, вкладок и состояний

Пройдись по каждому домену Master Prompt и убедись, что для него есть полноценный, а не пустой экран:

### Глобальная навигация
- Сайдбар/топбар с разделами: Дашборд, ИТ-Студия, Офис, Маркет, Лента, Чарты, Сообщество (чат), Уведомления, Профиль/Настройки, Админ (для Moderator/Admin).
- Индикатор текущей роли (Artist/Performer переключение, если у пользователя обе роли).
- Глобальный поиск.
- Счётчик непрочитанных уведомлений и сообщений чата.

### Дашборд
- Сводка активных проектов (студия/офис), статус фоновых AI-задач (progress bar, не просто текст), последние релизы, последние заказы на Маркете, последние уведомления.
- Пустое состояние для нового пользователя — с явным призывом к действию (создать первый проект), а не пустой белый экран.

### ИТ-Студия
- Список проектов (карточки: тема, жанр, настроение, статус, дата обновления).
- Экран создания: тема (счётчик до 200 символов), настроение, жанр, конструктор структуры.
- Двухпанельный редактор текста: текст + панель анализа (слоги, карта рифм, версии).
- Контекстный чат-ассистент как боковая панель, не модалка.
- Панель генерации аудио: статус задачи (PENDING/PROCESSING/SUCCEEDED/FAILED/CANCELLED) в виде понятного прогресса, а не сырого enum.
- Плеер с waveform для прослушивания демо.
- Кнопка «Отправить в Офис» с явным подтверждением и состоянием "уже отправлено".

### Офис
- Список проектов/релизов (Single/EP/Album) со статусами (`DRAFT_IN_OFFICE`, `VALIDATING`, `MODERATION`, `DELIVERED`, `RELEASED`, `REJECTED`, `FAILED`) как визуальный stepper, а не просто бейдж.
- Редактор метаданных релиза, треклист, contributors, explicit marker, даты релиза.
- Экран загрузки медиа с прогресс-баром multipart upload, обработкой разрыва сети и повторной попыткой.
- Отчёт AI Audio Scoring: BPM/key/spectral centroid/динамический диапазон/score/confidence — представь как читаемую карточку с объяснением, а не JSON.
- Статус дистрибуции (mock) с журналом попыток.

### Маркет
- Каталог кворков с категориями (дизайн/продакшн/звукорежиссура/сонграйтинг), фильтрами, поиском.
- Карточка кворка: портфолио, рейтинг, цена, защищённые media previews.
- Профиль исполнителя.
- Экран заказа: версии ТЗ, чат сделки, файлы, сдача результата, ревизии, приёмка.
- Экран спора (dispute): доказательства, история решений, статус.
- Личный кабинет: активные заказы (свои как заказчик и как исполнитель), баланс (hold/available), история транзакций (double-entry ledger в читаемом виде, не сырые записи).

### Лента
- Лента статей с категориями/тегами, пагинацией.
- Полноценная страница статьи с SEO-метаданными.
- Лайки, закладки, комментарии, счётчик просмотров.
- Админ/редакторская панель: черновики, публикация, модерация импортированных материалов.

### Чарты
- Гибридный чарт с историей позиций, индикатором динамики (рост/падение/new entry).
- Административная настройка весов источников (для Admin).

### Community Chat
- Список комнат, presence, typing indicator.
- Окно чата с историей, пагинацией, reconnect-индикатором ("переподключение…").
- Переход из чата в профиль пользователя и на его кворки.

### Уведомления, Профиль, Настройки
- Центр уведомлений с фильтрами по типу.
- Настройки профиля, ролей (Artist/Performer), сессий (журнал входов, logout одной/всех сессий), приватности.

### Админ-панель (Moderator/Admin)
- Модерация Ленты и импортированных материалов.
- Модерация споров Маркета.
- Настройка весов чартов.
- Просмотр audit log.

## Требования к состояниям (для каждого экрана из списка выше)

Для каждого экрана обязательно реализовать все четыре состояния, а не только "happy path":
1. **Loading** — скелетоны/спиннеры, соответствующие layout контента, а не generic spinner по центру.
2. **Empty** — осмысленный текст + призыв к действию, написанный от лица интерфейса ("Заполни настроение и жанр, чтобы начать" вместо "No data").
3. **Error** — конкретное объяснение, что произошло, и что делать дальше, без общих фраз "Something went wrong".
4. **Success/Populated** — полноценное отображение реальных данных с backend.

## Требования к качеству реализации

- WCAG 2.1 AA: видимый keyboard focus, контраст, семантическая разметка.
- Responsive от мобильного до desktop.
- Dark/light theme на основе токенов из `docs/DESIGN_SYSTEM.md`, а не хардкод цветов в компонентах.
- i18n (русский/английский) — весь новый текст интерфейса через ключи локализации, без хардкода строк.
- Анимации — точечно и обоснованно (переходы состояний задач, hover на карточках), не сплошной decorative motion.
- Никакой логики валидации только на фронтенде — если поле валидируется, backend-проверка обязана существовать (сверься с Master Prompt разделом 24).

## Протокол выполнения

1. Составь и зафиксируй `docs/DESIGN_SYSTEM.md` (палитра, типографика, layout-концепция, сигнатурный элемент) — с самокритикой плана перед стартом кодирования.
2. Пройдись по списку экранов выше и сопоставь с текущим состоянием `apps/web`: для каждого экрана зафиксируй в `docs/ACCEPTANCE_MATRIX.md` (или создай отдельный `docs/UI_AUDIT.md`, если matrix уже используется для другого) статус: отсутствует / есть, но пустой / есть частично / полностью готов.
3. Реализуй экраны по приоритету: сначала те, что уже частично готовы и не хватает состояний/данных, затем полностью отсутствующие.
4. Для каждого экрана: реализация UI → подключение к реальному API (не мок-данные в компоненте) → все 4 состояния → responsive → a11y → i18n-ключи.
5. После каждого экрана: линтер, type-check, визуальная проверка (скриншот, если доступно), обновление `docs/UI_AUDIT.md`.
6. По завершении всех экранов — полный проход по списку ещё раз, чтобы убедиться, что ничего не пропущено, и обновить `docs/IMPLEMENTATION_STATE.md`.

Не сообщай о завершении, пока каждый экран из раздела «Обязательный набор экранов» не имеет статус «полностью готов» с рабочей backend-интеграцией и всеми состояниями.
